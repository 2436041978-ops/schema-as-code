# Intent Steward 手册

> 面向域级自治负责人的实操指南。  
> 职责边界：在联邦宪法（`meta-rules.md`）框架内，拥有本域意图协议的立法权、解释权与废止权。

---

## 一、角色定义：你是谁

| 维度 | 定义 |
|------|------|
| **官方头衔** | Intent Steward（意图管家） / 域语义负责人 |
| **汇报对象** | 域 TL（技术负责人）+ 联邦治理委员会（架构契约层） |
| **服务对象** | 本域前端 / 设计 / AI 工程师（黑盒接入方） |
| **权力来源** | 联邦宪法 `meta-rules.md` 授予的**域级自治权** |
| **任期性质** | 兼职，通常由域内设计系统负责人或前端架构师兼任 |

**一句话定位**：你是本域的"语义立法者"——负责定义"这个域里，告警卡片长什么样、AI 能说什么、不能说什么"。

## 二、核心职责：三件事

### 2.1 意图协议立法（Legislation）

- 维护本域 `schema/` 目录下的 YAML 协议文件
- 定义本域专属的语义令牌（Semantic Token）、意图契约（Intent Contract）、同义词防火墙
- 确保新增意图与联邦宪法不冲突（命名空间、不可变边界、版本兼容性）

### 2.2 约束规则解释（Adjudication）

- 当业务工程师的代码被 Validator BLOCK 时，负责解释"为什么这条规则存在"
- 判定漂移事件属于"规则过松"还是"业务违规"
- 对沙盒域（`sandbox/`）内的豁免申请进行初审

### 2.3 版本治理与迁移（Governance）

- 管理本域意图协议的 SemVer 版本
- 评估 Breaking Change 对本域产品的影响面
- 组织本域内的渐进式迁移（而非全量强制升级）

## 三、工作流：从需求到协议

### 3.1 标准工作流（Happy Path）

```
业务方提出需求（如"新增一个支付确认弹窗"）
        │
        ▼
Steward 提取语义意图（"支付确认 = 资金操作 + 必须二次确认"）
        │
        ▼
查阅联邦宪法（检查命名冲突、边界兼容性）
        │
        ▼
编写 YAML 协议（语义令牌 + 意图契约 + 场景测试）
        │
        ▼
提交 PR 至本域 schema 仓库（附影响面分析）
        │
        ▼
域 TL 审批（技术可行性）
        │
        ▼
联邦委员会备案（全局命名空间注册）
        │
        ▼
Compiler 自动编译 → 产物分发至本域工程
        │
        ▼
业务方黑盒接入（按《前端黑盒接入指南》打标签即可）
```

### 3.2 漂移事件处理流（Exception Path）

```
Bridge 上报漂移事件（"本域 LLM 输出频繁触发 synonym-blacklist"）
        │
        ▼
Steward 归因分析（根因定位：规则过松？LLM 参数？业务场景变化？）
        │
        ▼
┌─────────────────┬─────────────────┐
│   规则过松       │   业务场景变化    │
│   → 收紧 YAML   │   → 扩展 allowed │
│   → 发 Patch    │   → 发 Minor     │
│     版本        │     版本         │
└─────────────────┴─────────────────┘
        │
        ▼
自动 PR 更新协议（Bridge 触发）或手工提交 PR
        │
        ▼
通知本域业务方（"下周一起，同义词规则收紧，请检查 Prompt 模板"）
```

## 四、权限矩阵：能做什么、不能做什么

| 操作 | Steward 权限 | 需升级至联邦委员会？ | 说明 |
|------|------------|---------------------|------|
| 新增本域语义令牌 | ✅ 自治 | 否 | 命名空间在 `domain.{本域}.*` 内 |
| 修改 `immutable: true` 令牌 | ❌ 禁止 | — | 必须发新版本，旧版冻结 |
| 新增跨域引用（如引用其他域令牌） | ⚠️ 申请 | 是 | 需委员会协调版本兼容性 |
| 修改联邦元规则（`meta-rules.md`） | ❌ 禁止 | — | 仅委员会有写入权 |
| 申请沙盒豁免（临时绕过某条规则） | ✅ 初审 | 是（终审） | 沙盒期最长 2 周 |
| 废止本域意图契约 | ✅ 自治 | 备案即可 | 需附迁移方案与替代协议 |
| 调整四层推演优先级 | ❌ 禁止 | — | 平台层统一配置 |

## 五、实操模板

### 5.1 意图协议编写检查清单

在提交 PR 前，逐条核对：

- [ ] 语义令牌命名符合 `domain.category.name` 三段式（如 `payment.confirmation.modal`）
- [ ] 意图契约已绑定至少一个语义令牌
- [ ] 不可变边界已明确 `violation_action`（block / escalate / fallback）
- [ ] 同义词防火墙已覆盖本域高频漂移词（至少 3 个 Edge Case）
- [ ] 场景测试包含：1 个 Happy Path + 2 个 Edge Case + 1 个 Fallback
- [ ] 版本号已按 SemVer  bump（Breaking → Major，新增 → Minor，收紧 → Patch）
- [ ] 影响面分析已列出下游产品/组件（避免静默破坏）

### 5.2 评审模板（PR Description）

```markdown
## 意图协议变更评审

**变更域**：payment
**变更类型**：Minor（新增支付确认意图）
**版本**：v1.2.0 → v1.3.0

### 新增
- 语义令牌：`payment.confirmation.modal`
- 意图契约：`payment-destructive-action`

### 影响面
- 下游产品：收银台、企业钱包、跨境支付
- 组件影响：ConfirmModal、PaymentButton
- 预计迁移成本：0（新增意图，无 Breaking Change）

### 测试
- [x] Happy Path：正常支付确认流程
- [x] Edge Case：LLM 建议"自动扣款" → 预期 BLOCK
- [x] Edge Case：缺少 `human_confirmed` → 预期 BLOCK

### 兼容性
- 与联邦宪法 `meta-rules.md` v2.1.0 无冲突
- 未引用跨域令牌
```

### 5.3 常见问题速查

**Q1：业务方说"这条规则太严，影响效率"，怎么办？**
> 先查沙盒豁免通道。如果属于短期实验，可申请 2 周豁免；如果属于长期需求，需评估是否放宽边界会引入系统性风险。Steward 有权在域内微调 `confidence_threshold`，但不可移除 `immutable_boundaries`。

**Q2：两个域都想定义 `status.critical`，命名冲突怎么办？**
> 联邦宪法规定：`status.critical` 是全局令牌，由委员会统一定义。本域应使用 `payment.status.critical` 或引用全局令牌，不可重复定义。

**Q3：Compiler 编译失败，但 YAML 看起来没问题？**
> 检查引用闭环：意图契约 → 语义令牌 → 约束规则 → 场景测试，是否有断链。常见错误：`rule_ref` 指向了不存在的文件路径。

**Q4：Bridge 上报大量漂移，但业务方否认改了 Prompt？**
> 查看 Bridge 归因报告中的 `llm_temperature` 与 `top_p` 字段。温度参数升高是同义词漂移的首要根因，建议联系 AI 工程师调整模型参数，而非直接放宽规则。

## 六、与平台团队的关系

| 事项 | Steward（域级） | 平台团队（联邦级） | 协作方式 |
|------|----------------|------------------|---------|
| 编写 YAML 协议 | ✅ 主责 | ❌ 不参与 | Steward 自治 |
| Compiler 插件开发 | ❌ 不参与 | ✅ 主责 | 平台提供，Steward 消费 |
| Validator 规则引擎升级 | ❌ 不参与 | ✅ 主责 | 平台统一升级，Steward 感知 |
| Runtime 守卫框架 | ❌ 不参与 | ✅ 主责 | 平台提供 SDK，Steward 配置绑定 |
| 跨域命名空间仲裁 | ⚠️ 申请 | ✅ 裁决 | 委员会协调 |
| 沙盒豁免终审 | ⚠️ 初审 | ✅ 终审 | Steward 提申请，委员会 24h 内批复 |

**沟通协议**：
- 日常问题：在 `#intent-steward` 频道提问，平台团队 4h 内响应
- 紧急 BLOCK（影响生产）：直接 `@platform-oncall`，1h 内响应
- 季度同步：Steward 参加联邦委员会架构 Review，汇报本域规则健康度

## 七、结语：域级自治不是各自为政

Intent Steward 的权力来自联邦宪法的**授权**，而非**让渡**。你的自治边界是清晰的：

- **域内**：你可以快速立法、快速迭代、快速响应业务需求，无需每次审批
- **域外**：你必须遵守命名空间、版本契约、不可变边界，确保联邦网格的整体一致性

好的 Steward 不是"规则的看守者"，而是"意图的翻译官"——把业务需求翻译为机器可读的契约，让工程师在不知情的情况下，就已经走在合规的轨道上。
