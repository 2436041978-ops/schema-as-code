# 联邦治理拓扑

> Schema-As-Code 联邦自治架构的全景视图——从控制平面到数据平面，从意图声明到运行时拦截。

---

## 一、核心隐喻：联邦制国家

Schema-As-Code 的治理模型借鉴**联邦制（Federalism）**的宪政结构：

- **中央（控制平面）**：制定统一语义宪法（`intent-schema-compiler`），定义不可变边界、语义令牌、人机契约。
- **各州（数据平面）**：在联邦宪法框架下自治运行，通过 Compiler/Validator/Runtime/Bridge 执行本域规则。
- **宪法法院（Registry）**：裁决规则冲突、管理版本兼容性、审查 Breaking Change。
- **监察机构（Bridge）**：观测各州运行时漂移，提出宪法修正案。

联邦制的核心优势在于**统一而不集权**——语义标准全局一致，但执行策略允许域级自治。

- **横向轴**：语义治理网格（控制平面+数据平面）——承载规则与观测的流动，正交穿透所有纵向层级。

## 三、控制平面：联邦宪法

控制平面是**单一事实源（Single Source of Truth）**，所有设计意图的宪法文本。

### 3.1 三层协议结构

| 层级 | 宪法职能 | 文件形态 | 稳定性 |
|------|---------|---------|--------|
| **语义层** | 定义联邦的"通用语言"——语义令牌、意图契约、同义词防火墙 | `semantic-tokens.yaml` | 高（`immutable: true`） |
| **治理层** | 定义联邦的"禁止性法律"——人机边界、输入输出约束、安全规则 | `human-ai-boundary.yaml` | 中 |
| **执行层** | 定义联邦的"司法程序"——编译思维链、场景测试用例 | `scenario-tests.yaml` | 低（随场景迭代） |

### 3.2 版本化与修宪机制

- **语义版本（SemVer）**：`v1.0.0` 的语义令牌一旦发布，冻结不可变；变更必须发新版本。
- **Git-Native 修宪**：任何规则变更 = Git PR → 影响面分析 → 委员会审批 → Merge → 自动编译分发。
- **Breaking Change 公投**：修改 `immutable: true` 的令牌需联邦治理委员会审批，并触发下游全量重编译。

## 四、数据平面：五模块联邦执行机构

数据平面是联邦宪法的**分布式执行机构**，五模块各自独立运行，通过标准接口协议通信。

### 4.2 模块职责与接口

| 模块 | 联邦角色 | 核心职责 | 接口形态 |
|------|---------|---------|---------|
| **Registry** | 宪法档案馆 | 语义 ID 全局注册、版本管理、跨域影响面分析 | Git 仓库 + YAML 索引 |
| **Compiler** | 司法解释机构 | 将 YAML 宪法翻译为各层可执行规则（TS/ESLint/OpenAPI/OPA） | npm CLI + 插件化产物 |
| **Validator** | 联邦法院 | 对 LLM 输出/组件 Props/API 响应执行四层推演安检 | npm CLI + YAML 规则库 |
| **Runtime** | 联邦警察 | 在组件渲染、API 调用、LLM Tool 执行时现场拦截 | npm 包（HOC/中间件/Tool包装器） |
| **Bridge** | 监察委员会 | 采集运行时漂移，归因到宪法条款，自动提出修正案 | GitHub Action + 归因引擎 |

## 五、五层穿透模型：纵向联邦管辖

联邦宪法不仅横向管理，还要纵向穿透所有业务层级：

| 层级 | 联邦管辖范围 | 编译产物 | 拦截时机 |
|------|-------------|---------|---------|
| **L1 Token 层** | 视觉语义统一 | CSS 变量 + 语义注释 | 主题编译时 |
| **L2 组件层** | 组件语义绑定 | TS 类型 + Prop 约束 + ESLint 规则 | 开发编译时 + 组件渲染时 |
| **L3 API 层** | 接口语义契约 | OpenAPI 扩展 + JSON Schema | 请求到达业务逻辑前 |
| **L4 容器层** | 服务语义策略 | OPA Policy / WASM 配置 | 网络请求路由时 |
| **L5 数据库层** | 数据语义标签 | DDL 注释 + CHECK 约束 | 数据写入时（软约束） |

**穿透原则**：
1. **同构映射**：同一语义令牌（如 `status.critical`）在五层中保持语义一致性。
2. **增量管辖**：新增约束时，Compiler 只生成对应层级的增量产物，Registry 定位需更新的下游节点。
3. **双向溯源**：任意层级的约束产物携带 `intent_id` 与 `schema_version_hash`，支持从运行时拦截事件反向定位到宪法源条款。

## 六、联邦自治原则

### 6.1 中央与地方的权力边界

| 权力类型 | 控制平面（中央） | 数据平面（地方/域级） |
|---------|----------------|---------------------|
| **立法权** | 定义语义令牌、意图契约、元规则 | 定义域级适配规则、沙盒实验规则 |
| **司法权** | 裁决 Breaking Change、版本兼容性 | 执行四层推演、现场拦截 |
| **监察权** | 制定治理指标、组织级仪表盘 | 采集域级漂移事件、提出修宪建议 |
| **豁免权** | 无（宪法不可豁免） | 沙盒域（`experimental/`）可临时豁免非安全约束 |

### 6.2 域级自治（Domain Autonomy）

- **Intent Steward**：每个业务域指定一名语义管家，负责维护本域的 `bindings/domain-x.yaml`（引用哪些意图契约）。
- **沙盒域**：`schema/experimental/` 允许未注册的新意图快速迭代，成熟后通过 PR 晋升到正式域。
- **适配器自治**：各域可独立开发 Compiler 插件（如内部组件库适配），向联邦注册表报备即可。

## 七、与现有生态的联邦关系

Schema-As-Code 联邦不替代任何现有主权国家（技术栈），只与其建立**外交条约**：

| 现有生态 | 联邦关系 | 条约内容 |
|---------|---------|---------|
| **Ant Design / Carbon** | 设计系统同盟 | 联邦向组件库注入语义注释（Token 元数据），组件库向联邦提供 Props 拦截接口 |
| **LoongSuite GenAI SemConv** | 可观测同盟 | 联邦将拦截事件写入 LoongSuite Trace；LoongSuite 将漂移归因反向驱动联邦修宪 |
| **DESIGN.md** | 设计表达同盟 | 联邦从 DESIGN.md 提取语义令牌；DESIGN.md 从联邦获取"AI 注意事项"约束 |
| **OpenTelemetry** | 观测基础设施同盟 | Bridge 通过 OTel 协议采集 Trace，不依赖特定云厂商 |

---

## 八、结语：拓扑即治理

联邦治理拓扑不是一张静态架构图，而是**动态的权力分配协议**：

- 控制平面掌握**立法权**（定义什么是对的）
- 数据平面掌握**执法权**（执行和拦截）
- Bridge 掌握**监察权**（发现和修正）
- 各业务域掌握**自治权**（在宪法框架内灵活适配）

当这套拓扑运转起来时，设计意图治理就从"依靠人的理解和记忆"升级为"依靠系统的编译和执行"——这就是 Schema-As-Code 联邦自治的宪政目标。
