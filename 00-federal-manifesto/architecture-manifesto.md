# Schema-As-Code 架构宣言与核心原则

> 本文是 Schema-As-Code 联邦自治架构的宪法级文档。它定义了这套工程方法论的元规则——不是"怎么做"，而是"什么不能做"以及"为什么必须这样做"。

---

## 一、宣言：从意图断裂到意图宪法

在《设计意图治理》系列的前三篇中，我们论证了一个结构性困境：

- **语义断裂**：同一概念在不同产品中定义不一致
- **规则断裂**：约束以文档形态存在，无法被自动执行
- **验证断裂**：缺乏自动化验证手段，依赖人肉走查

当 LLM 加入界面生产后，这三重断裂被指数放大。确定性界面的治理已经很难，概率性界面的治理连"人查"都做不到——因为 LLM 默认不会主动遵循设计规范，除非规范被显式编码到 Prompt 或工具调用中。

**Schema-As-Code 的提出，正是为了建立一份"意图宪法"**——让设计意图从"依靠人的理解和记忆"升级为"依靠系统的编译和执行"。

这份宪法不替代任何现有技术栈（Ant Design、Carbon、LoongSuite、OpenTelemetry），它只填补一个空白：**语义契约层**——在"视觉组件"和"运行时观测"之间，建立一套可版本化、可编译、可拦截、可观测的意图协议。

---

## 二、核心原则（The Seven Principles）

### 原则 1：协议即代码（Protocol as Code）

设计意图必须以机器可读的格式（YAML/JSON）存储于版本控制系统中，具备完整的 Git 历史、Diff 可见性、影响面分析和自动编译能力。

**反模式**：意图定义写在语雀文档、飞书表格或口头约定中。

### 原则 2：声明式治理（Declarative Governance）

治理系统只需声明"意图应该是什么样子"，由编译器和运行时自动收敛到该状态。工程师不需要手动同步规范，只需要消费编译产物。

**反模式**：每次规范更新后，手动通知 10 个前端负责人修改代码。

### 原则 3：正交穿透（Orthogonal Penetration）

语义治理网格横向穿透所有纵向生产链路（Token → 组件 → API → 容器 → DB / LLM），不替代任何现有技术栈，只向其注入规则与采集观测。

**反模式**：为了统一语义，强制所有产品迁移到同一套组件库。

### 原则 4：联邦自治（Federal Autonomy）

中央平台制定元规则（语义注册表、编译协议、校验接口），各业务域（Domain）在元规则框架内自主立法（定义域级语义令牌、意图契约、豁免规则）。中央不干预域内细节，只确保跨域语义一致性。

**反模式**：中央团队统一定义所有产品的所有语义，导致僵化与创新受阻。

### 原则 5：约束显化（Constraint Explicitation）

所有不可突破边界必须以结构化格式显式声明，禁止隐含假设。每条约束必须携带：边界类型（语义/安全/美学/合规）、违规动作（block/escalate/fallback）、场景测试（Happy Path + Edge Case）。

**反模式**："高危操作必须二次确认"写在设计规范里，但代码中没有强制校验。

### 原则 6：阻断优于修正（Block over Retry）

校验失败时，禁止触发 LLM 自动重试或自动修正。自动重试会引入新的概率漂移，破坏语义确定性。正确行为是：阻断交付 → 记录日志 → 升级人工。

**反模式**：Validator 发现违规后，让 LLM "重新生成一次"， hoping for the best.

### 原则 7：观测反哺约束（Observability Feeds Governance）

运行时发现的语义漂移案例必须反向驱动设计时规则的迭代。Bridge 采集的漂移事件应自动归因到具体契约规则，并创建 Registry PR 收紧约束。

**反模式**：运行时观测只用于告警和排障，不回流到设计规范。

## 三、联邦自治治理模型

Schema-As-Code 采用联邦制（Federalism）而非中央集权制。原因很直接：

- 中央团队不可能理解所有业务域的语义细节
- 过度中央化会导致治理僵化，产品创新受阻
- 但完全自治会导致跨域语义断裂，回到治理熵增

### 三层治理结构

```
┌─────────────────────────────────────────┐
│  联邦层（Federal Layer）                  │
│  ─────────────────────────────────────   │
│  中央平台团队维护：                        │
│  • 元规则清单（Meta-Rules）               │
│  • 语义注册表（Registry）                 │
│  • 编译协议（Compiler Spec）              │
│  • 校验接口（Validator API）              │
│  • 观测协议（Bridge Protocol）            │
└─────────────────────────────────────────┘
                    │
                    ▼ 元规则约束
┌─────────────────────────────────────────┐
│  域级层（Domain Layer）                   │
│  ─────────────────────────────────────   │
│  各业务域 Intent Steward 维护：            │
│  • 域级语义令牌（Domain Tokens）          │
│  • 域级意图契约（Domain Contracts）       │
│  • 域级豁免规则（Domain Exemptions）       │
│  • 域级适配器（Domain Adapters）          │
└─────────────────────────────────────────┘
                    │
                    ▼ 编译产物
┌─────────────────────────────────────────┐
│  业务层（Business Layer）                 │
│  ─────────────────────────────────────   │
│  前端/设计/AI 工程师消费：                 │
│  • 编译后的 ESLint 规则                   │
│  • 编译后的 TS 类型定义                   │
│  • 编译后的 OpenAPI 扩展                  │
│  • 运行时 HOC / 中间件 / Tool Guard       │
└─────────────────────────────────────────┘
```

### 中央与域的权责边界

| 维度 | 联邦层（中央） | 域级层（业务域） |
|:---|:---|:---|
| **语义 ID 命名空间** | 定义全局命名规范（`domain.category.name`） | 在规范内注册具体令牌 |
| **不可变规则** | 定义安全/合规类全局不可变边界 | 定义域内业务语义边界 |
| **版本协议** | 制定 SemVer 与 Breaking Change 规则 | 执行版本发布与影响面分析 |
| **编译器** | 提供编译器核心与插件框架 | 开发域级适配器插件 |
| **沙盒豁免** | 制定豁免申请与审批流程 | 在沙盒域内快速迭代实验性语义 |

## 四、与现有体系的互补边界

Schema-As-Code 不替代任何现有工具，只填补它们之间的语义断层。

| 现有方案 | 解决什么问题 | 未解决什么问题 | 本架构的填补位置 |
|:---|:---|:---|:---|
| **Ant Design / Carbon** | 视觉属性同步 | 视觉属性的业务语义定义及 LLM 约束 | Token 层之上的语义契约层 |
| **OpenAI Structured Outputs** | LLM 输出格式 100% 合法 | 输出内容的业务语义正确性 | 格式校验之上的语义推演层 |
| **LoongSuite GenAI SemConv** | AI 运行时观测 | 漂移发生前如何约束预防 | 运行时观测之上的设计时约束层 |
| **Policy-as-Code (OPA)** | 安全权限控制 | 体验语义与人机边界 | 权限策略之上的意图语义层 |
| **JSON Schema / Protobuf** | 数据结构定义 | 设计语义定义、校验失败无降级策略 | 工程师 Schema 之上的设计语义 Schema |
| **DESIGN.md** | 设计意图的丰富表达 | 意图的刚性约束与机器执行 | 描述层之下的约束编译层 |

**互补产生组织经济学价值**：
- DESIGN.md 让 AI 看清目标（正向表达）
- Schema-As-Code 让 AI 锁死边界（负向约束）
- LoongSuite 让漂移被发现（运行时观测）
- Bridge 让规则被修正（闭环反哺）

四者结合，构成完整的 AI 界面治理工作流。

## 五、定义边界：Schema-As-Code 不是什么

为了进一步精确这个概念，明确以下边界：

| 邻近概念 | Schema-As-Code 的区别 |
|:---|:---|
| **Design System（Ant Design / Carbon）** | 不替代组件库，只在 Token/组件之上增加语义契约层 |
| **Prompt Engineering** | 不是临时调优 Prompt，而是将意图边界编译进系统 |
| **Policy-as-Code（OPA）** | 不替代安全策略，只在权限之上增加体验语义层 |
| **JSON Schema / Protobuf** | 不是数据结构定义，而是设计语义定义 |
| **LLM Guardrails（商业产品）** | 不是黑盒服务，而是 Git-Native、可审计、可版本化的开放协议 |
| **Low-code / No-code 平台** | 不是让非工程师写代码，而是让工程师用代码治理设计意图 |

## 六、结语：立宪是为了自治

Schema-As-Code 的终极目标不是中央控制，而是**联邦自治**——通过一套清晰的元规则，让各业务域在互不干扰的前提下自主演进，同时保证跨域语义一致性。

这份宣言定义了元规则的元规则（Meta-Meta-Rules）。它足够轻量，可以在 5 分钟内读完；它足够刚性，可以作为后续所有架构决策的上诉法庭。

当组织中的每一个设计师、前端工程师、AI 工程师都能通过阅读 YAML 文件理解设计意图的边界时，当每一次语义变更都能通过 Git Diff 自动传导到所有下游产物时，当每一次 LLM 漂移都能被运行时拦截并反向修正协议时——治理就从成本中心变成了杠杆资产。

这就是 Schema-As-Code 的立宪意义。

## 附录：术语速查

| 术语 | 定义 |
|:---|:---|
| **Schema-As-Code** | 将设计意图的形式化契约以机器可读格式（YAML/JSON）编码于版本控制系统中，并通过编译器转化为可执行约束产物的工程方法论 |
| **声明式语义治理网格** | 以控制平面存储意图协议、以数据平面分布式执行约束与观测、通过 GitOps 闭环实现自动收敛的架构体系 |
| **联邦自治** | 中央平台制定元规则，各业务域在元规则框架内自主立法，通过语义注册表确保跨域一致性的治理模型 |
| **Intent Steward** | 业务域内负责维护域级语义令牌、意图契约与豁免规则的治理角色 |
| **约束显化** | 将隐含的设计约束转化为结构化、机器可读、可版本化的显式协议的过程 |
| **阻断优于修正** | 校验失败时不触发自动重试，直接阻断交付并升级人工的核心策略 |
| **观测反哺约束** | 运行时漂移事件反向驱动设计时规则迭代的闭环机制 |
