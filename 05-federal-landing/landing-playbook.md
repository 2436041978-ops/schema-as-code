# 联邦落地手册

> 本文档面向联邦治理委员会、平台团队负责人及各域技术 Lead，提供 Schema-As-Code 从试点到全域推广的渐进式落地路径、组织角色定义与风险治理框架。

---

## 一、手册定位

Schema-As-Code 联邦自治架构不是一次性建设项目，而是**组织级基础设施的持续运营过程**。本手册回答三个核心问题：

1. **谁来做**：组织内需要哪些角色，各自的权力边界与协作接口是什么？
2. **怎么做**：从 0 到 1 再到 N 的三阶段渐进路径，每阶段的准入条件与退出标准？
3. **怎么防**：规模化过程中的六大风险及对应的缓解策略？

## 二、组织角色与职责

### 2.1 联邦治理委员会（Federal Governance Committee）

| 属性 | 说明 |
|------|------|
| **组成** | 技术 VP / 架构委员会主席 / 设计系统负责人 / 安全合规代表 |
| **职责** | 审批联邦宪法（顶层架构设计）、裁定跨域语义冲突、批准 Breaking Change |
| **权力** | 对 `01-architecture-contract/` 目录的 Merge 拥有一票否决权 |
| **投入** | 每月 1 次评审会议（1 小时），紧急事项异步审批 |

### 2.2 语义架构师（Semantic Architect）

| 属性 | 说明 |
|------|------|
| **人数** | 1 人（可兼职，建议由设计系统负责人或前端架构师担任） |
| **职责** | 定义语义域划分（认知/交互/交易/观测）、审核 Breaking Change、维护 `meta-schema.json` |
| **权力** | 对 `00-federal-manifesto/` 及 `03-domain-autonomy/` 中的语义定义有审批权 |
| **投入** | 每周 4-6 小时 |

### 2.3 平台维护者（Platform Maintainers）

按模块拆分，每个模块 1 人兼职：

| 角色 | 模块 | 职责 | 投入 |
|------|------|------|------|
| **Registry 维护者** | 模块 1 | 版本管理脚本、CI 影响面分析、发布流水线 | 每周 2-3 小时 |
| **Compiler 维护者** | 模块 2 | 编译器核心管线、插件框架、产物发布 | 每周 3-4 小时 |
| **Validator + Runtime 维护者** | 模块 3 + 4 | 四层推演引擎、运行时守卫、策略热加载 | 每周 3-4 小时 |
| **Bridge 维护者** | 模块 5 | 采集适配器、归因引擎、闭环反哺 PR 创建 | 每周 2-3 小时 |

### 2.4 Intent Steward（域级语义代表）

| 属性 | 说明 |
|------|------|
| **人数** | 每业务域 1 人（前端 TL 或设计负责人兼职） |
| **职责** | 维护本域的 `bindings/*.yaml`（产品绑定配置）、在域内推广意图协议、处理本域的 BLOCK 事件 |
| **权力** | 对本域绑定文件有写权限；对本域沙盒语义域（`experimental/`）有自主审批权 |
| **投入** | 每周 1-2 小时 |

### 2.5 产品接入工程师（Product Integration Engineer）

| 属性 | 说明 |
|------|------|
| **人数** | 每产品 1 人（前端/后端工程师兼职） |
| **职责** | 在产品代码中植入 `withIntentGuard` 或 `apiGuardMiddleware`、配置 `x-actor-type` 请求头 |
| **投入** | 一次性 2-4 小时（接入期），后续按需维护 |

## 三、三阶段渐进落地路径

### Phase 1：语义锚定（1-2 个月）

**目标**：建立控制平面雏形，完成 Token→组件层的穿透，验证"约束显化"价值。

**准入条件**：
- 组织内并行产品数 ≥ 3
- 至少 1 个核心产品（如告警系统、支付系统）愿意试点
- 语义架构师人选确定

**具体动作**：

| 周次 | 动作 | 产出 | 负责角色 |
|------|------|------|---------|
| W1 | 搭建 `intent-schema-compiler` 语义仓库，定义首批语义令牌（`status.critical`, `action.destructive`） | 语义层 YAML v0.1.0 | 语义架构师 |
| W2 | 建立 Registry 版本化管理 + CI 影响面分析 | 版本化语义索引 + GitHub Action | Registry 维护者 |
| W3 | 实现 Compiler v0.1：Token Plugin + Component Plugin | 生成 Design Token 扩展 + TS 类型约束 | Compiler 维护者 |
| W4 | 在试点产品中接入 `withIntentGuard`（1 个核心组件） | 编译时自动报错演示 | 产品接入工程师 |
| W5-6 | 走通 1 个端到端场景（如告警卡片），验证 YAML → 编译 → 组件约束的链路 | 场景测试 PASS 报告 | 全角色协作 |

**退出标准**：
- 语义一致性从"人查"变为"编译时自动报错"
- 试点团队反馈：接入成本 < 4 小时，拦截价值 > 人工走查 2 周

**下阶段预告**：契约闭环阶段将引入 Validator 四层推演与 LLM 输出安检。

---

### Phase 2：契约闭环（2-3 个月）

**目标**：打通治理层与 LLM 层，实现输入约束注入 + 输出校验拦截，建立人机边界。

**准入条件**：
- Phase 1 退出标准达成
- 至少有 1 个 LLM 生成内容直接进界面的场景（如 AI 助手、智能客服）
- Validator 维护者到位

**具体动作**：

| 周次 | 动作 | 产出 | 负责角色 |
|------|------|------|---------|
| W1-2 | 扩展 Compiler：Prompt Plugin + API Plugin | Prompt 约束前缀 + OpenAPI 扩展片段 | Compiler 维护者 |
| W3-4 | 部署四层推演引擎：在 LLM 输出管道中集成语法/语义/安全推演 | Validator CLI 可运行 | Validator 维护者 |
| W5-6 | 建立人机边界运行时策略：高危操作强制阻断并升级人工 | Runtime HOC + API 中间件 | Runtime 维护者 |
| W7-8 | 接入 LoongSuite SemConv：将拦截事件写入 Trace，启动语义一致性指标采集 | 首批治理指标 | Bridge 维护者 |
| W9-10 | 推广至 2-3 个新产品，建立域级 Intent Steward 机制 | 多域绑定配置 | Intent Steward |

**退出标准**：
- LLM 生成内容进入界面前，必须经过 Validator "安检门"
- 人机边界拦截率 > 95%（高危操作无漏过）
- 语义一致性得分可量化（基线建立）

**下阶段预告**：网格治理阶段将扩展为多产品治理平台，实现全链路观测闭环。

---

### Phase 3：网格治理（3-6 个月）

**目标**：扩展为 5+ 产品的联邦治理平台，实现全链路观测闭环与组织级指标。

**准入条件**：
- Phase 2 退出标准达成
- 并行产品数 ≥ 5，或 LLM 消费场景 ≥ 10
- 联邦治理委员会正式成立

**具体动作**：

| 月次 | 动作 | 产出 | 负责角色 |
|------|------|------|---------|
| M1 | 推广 Registry 至 5+ 产品，建立跨产品语义注册表 | 全局语义一致性仪表盘 | Registry 维护者 + 委员会 |
| M2 | 实现 Schema 版本化的影响面分析：变更自动触发下游产品 CI 重编译 | 影响面报告自动化 | Compiler 维护者 |
| M3 | 部署容器层 Sidecar 拦截：API 网关和容器入口的语义策略执行 | 运行时拦截覆盖率 100% | Runtime 维护者 |
| M4-5 | 上线组织级治理仪表盘：实时展示语义一致性得分、规则拦截率、人工升级比例 | 治理健康度可视化 | Bridge 维护者 + 委员会 |
| M6 | 建立联邦宪法修订流程：Breaking Change 审批、版本淘汰、迁移窗口 | 治理运营手册 v1.0 | 委员会 |

**退出标准**：
- 设计意图治理从"单点工具"升级为"组织级基础设施"
- 治理健康度评分 ≥ 75 分（B 级）
- 人工升级比例 < 10%

## 四、决策流与协作流程

### 4.1 日常决策流（产品团队发现语义漂移）

```
产品团队发现语义漂移
    │
    ▼
提交 Issue 至 schema-as-code 仓库（附漂移案例）
    │
    ▼
Intent Steward 初审：是域内规则缺失还是规则过松？
    │
    ├──► 域内规则缺失 ──► Intent Steward 创建 YAML 定义 PR
    │                      └──► 语义架构师审批 ──► Merge ──► Compiler 自动分发
    │
    └──► 规则过松 ──► Bridge 自动创建收紧 PR（归因引擎定位根因）
                           └──► 语义架构师审批 ──► Merge ──► Compiler 自动分发
```

### 4.2 紧急决策流（安全边界突破）

```
Runtime 检测到 AI 越权执行 destructive-action
    │
    ▼
立即 BLOCK + 记录日志 + 升级人工（分钟级）
    │
    ▼
Intent Steward 2 小时内提交紧急补丁 PR
    │
    ▼
语义架构师异步审批（4 小时内）
    │
    ▼
Merge 后触发热加载（无需发版，下一事件循环生效）
    │
    ▼
Bridge 记录拦截事件，纳入月度治理报告
```

### 4.3 跨域冲突裁决流

```
域 A 定义 "红色" = 告警
域 B 定义 "红色" = 促销强调
    │
    ▼
Intent Steward 双方提交语义注册申请
    │
    ▼
联邦治理委员会裁定：
    ├──► 若冲突不可调和 ──► 强制 namespace 隔离（domain-a.red / domain-b.red）
    └──► 若可调和 ──► 升级语义粒度（status.critical vs action.promotion）
```

## 五、风险矩阵与缓解策略

| 风险 | 描述 | 概率 | 影响 | 缓解措施 | 负责角色 |
|------|------|------|------|---------|---------|
| **治理僵化** | 过度严格的语义锁定导致产品创新受阻 | 高 | 高 | **沙盒语义域**：`experimental/` 允许未注册意图快速迭代，成熟后晋升 | 语义架构师 |
| **绕过风险** | 开发者不接入 `withIntentGuard`，直接调用底层 API | 高 | 高 | **网关兜底** + **代码扫描审计**：每月检测未绑定意图的组件/API 调用 | Runtime 维护者 |
| **误报率过高** | 规则过于严格，大量合法输出被 BLOCK | 高 | 高 | **灰度规则**：新规则先以 `warn` 模式运行 2 周，验证后再升级 `block` | Validator 维护者 |
| **与现有平台耦合** | 公司切换日志/Trace 系统导致 Bridge 失效 | 低 | 中 | **采集器抽象层**：每个平台一个适配器文件；优先消费 OTel 标准格式 | Bridge 维护者 |
| **闭环延迟** | 从漂移发现到规则生效全链路耗时 1-2 周 | 高 | 中 | **双速闭环**：紧急走人工通道（小时级）；常规走自动 PR（周级）；Runtime 支持热补丁 | 委员会 |
| **产物版本漂移** | 产品加载旧版策略，与 Registry 最新契约不一致 | 中 | 高 | **版本哈希校验**：Runtime 启动时校验策略文件哈希，不匹配则告警 | Registry 维护者 |
| **组织采纳阻力** | 前端团队觉得"多一层约束降低效率" | 高 | 高 | **渐进式接入**：先只对接 1 个核心产品 + 1 个核心意图；用 CI 自动拦截代替人工审查 | Intent Steward |
| **单点瓶颈** | 语义架构师成为瓶颈，所有语义定义都要审批 | 中 | 中 | **语义域自治**：按业务线拆分审批权，架构师只审跨域变更 | 委员会 |

## 六、ROI 与组织经济学价值

### 6.1 成本结构对比

| 成本项 | 未治理场景（人工） | 治理后场景（Schema-As-Code） |
|------|------------------|---------------------------|
| **走查成本** | 50 产品 × 500 页面 × 人工覆盖率 20% | 四层推演校验覆盖率 100%，人工仅处理 BLOCK 事件 |
| **回滚成本** | 语义漂移导致用户误操作，事后修复 + 舆情处理 | 输出侧自动拦截，漂移不进入生产环境 |
| **对齐成本** | 新规范发布 → 10 个负责人手动同步 → 遗漏概率指数增长 | 意图协议版本化更新 → Git Diff 自动触发影响面分析 → 下游自动重编译 |
| **认知成本** | 运维工程师在 3 个产品间切换，同一颜色语义不同 | 语义令牌全局统一，跨产品认知一致性由系统保证 |

### 6.2 盈亏平衡点

> **当 N > 5（并行产品超过 5 个）、K > 10（LLM 消费场景超过 10 个）时，未治理系统的熵增成本将首次超过建立治理体系的固定投入成本。此时治理从"成本中心"转变为"负成本操作"——不治理的代价更高。**

### 6.3 治理杠杆公式

```
治理杠杆 = (人工走查成本 - 自动校验成本) × 产品数 × 时间跨度
         ──────────────────────────────────────────────────────
                    治理体系建设固定投入

当 治理杠杆 > 1 时，Schema-As-Code 产生正向 ROI
当 治理杠杆 > 3 时，Schema-As-Code 成为组织级战略资产
```

## 七、版本管理与迁移策略（摘要）

详细规范见：[版本管理与迁移拓扑](./version-migration.md)

### 7.1 语义版本规范（SemVer）

| 版本变更 | 判定标准 | 对下游影响 | 迁移策略 |
|---------|---------|-----------|---------|
| **Major（X.0.0）** | 不可变令牌被修改、语义域重新划分 | Breaking Change，旧产物冻结 | 手动升级，给予 4 周迁移窗口 |
| **Minor（x.Y.0）** | 新增语义令牌或意图契约 | 向下兼容，旧消费者不受影响 | 自动分发，无需人工干预 |
| **Patch（x.x.Z）** | 规则收紧、同义词扩展、描述优化 | 产物自动热更新 | CI 自动触发，秒级生效 |

### 7.2 版本淘汰策略

- 只保留最近 **3 个 Major 版本** 的完整产物
- 旧版本进入 **只读归档**（archive），不删除，供审计追溯
- `latest` 软链始终指向最新稳定版本，但生产环境建议显式锁定版本号

## 八、结语：从项目到基础设施

Schema-As-Code 联邦自治架构的落地不是"上线一个系统"，而是**建立一种组织习惯**：

- **设计师**习惯用 YAML 定义语义边界，而非用文档描述规范
- **工程师**习惯在编译时看到红色报错，而非在走查时听到设计师的反馈
- **架构师**习惯用 Git Diff 审计变更，而非用会议同步决策
- **管理者**习惯用治理仪表盘衡量健康度，而非用事故数量评估风险

当这种习惯成为默认工作方式时，语义治理就从"额外负担"变成"空气"——无处不在，却几乎无感知。
